## 二、经验和教训

### 决策原则

1. **分工判断**：功能重叠时问"产出是决策型还是执行型？"
2. **泳道图优于流程图**：多角色多情景用一张图展示所有流转路径
3. **用户隐含需求**：重视质控（审核产出）+ 闭环确认（反问）+ 一次聊透（穷举）；不擅自抹掉用户原始语义
4. **工具选择决策树**：能调工具不动脚本，能动脚本不动架构。规则明确+纯机械转换（格式迁移/批量替换）→ 用10-30行Python脚本零幻觉；prompt级/工具链能解决 → 不动架构，一次输出连续调多个工具。LLM在大数据量格式化任务中易漏行、成本高
5. **信息损耗认知**：多Agent管道中每经文本转换丢失像素级信息。端到端让VLM原始输出直接喂给下游Agent，不走中间文本转译
6. **冗余设计价值**：角色分工是"结构上的不可替代"提供 check-and-balance 质量闸门，非"功能上的不可替代"
7. **层级不对齐逐层确认**：数据结构与模板层级数不对齐时，禁止猜测映射关系 → 画两侧层级结构 → 表格化标注对应关系 → 提交用户确认后再编码
8. **关键设计决策写死**：产品文档中写死核心决策+理由（建议表格化），后续对齐有据可查，防止反复争论
9. **研发反馈二阶判断**：研发说"不需要关心这个场景" → 先判是技术简化还是需求裁剪。技术方案使产品侧业务逻辑完全一致 → 合并采纳；仍有用户可见差异 → 保留区分
10. **Skill解耦设计原则**：设计多Agent协作Skill时，优先拆成独立职责单元（非流水线步骤），中间留人工决策节点。判断标准："这些步骤必须自动串联吗？还是可以让人在中间做决策？"——PM角色是审核和决策时，不应设计自动流水线。
11. **规范集中化设计**：AI产出的代码应设单一修改点——所有可配置的规范（风格/结构/命名）集中一处。前端反馈→改规范→重跑→验证，形成闭环，无需改代码生成逻辑。适用于任何需与外部团队协作的AI代码产出场景。
12. **React项目优于HTML全量重写**：Vite+React+TS+Antd+ECharts 项目结构下，增改组件比整页HTML重写省约70% Token。
13. **技术决策解释框架（面向非技术角色）**：用「对用户/业务的影响」作为第一层解释，技术细节作为第二层。给PM解释前端决策时锚定"加载速度/维护成本→交付速度"这条可见影响链。

### 执行规范

14. **七仔权限分层**：流程性可代签（文档完整/需求清晰/复杂度判断），技术性不碰（选型/架构/代码质量）。**重要补充**：七仔角色是"启示器/头脑风暴伙伴"，讨论方案阶段克制"我来帮你做"的本能冲动。只有用户明确说"现在开始做"或"你去执行"才动手；用户说"不要着急干活"时立即退回方案讨论模式。
15. **知识萃取零术语继承**：输出禁止引用来源文章的术语/名词/框架名。每条锚定无需背景即可识别的现象。自检三问：含术语？→ 重写；首句抽象？→ 重写；路人看不懂？→ 重写
16. **去AI味四层检测**：词汇层（术语密度）→ 句式层（排比/反问问答频率）→ 结构层（类比堆叠，同一段≥2个类比即AI味）→ 姿态层（过度自省/讨好式总结），姿态层最隐蔽最致命
17. **反馈与验证双检机制**：收到执行反馈 → 先查skill/规范是否已覆盖该场景。已覆盖是执行问题改执行，未覆盖是规范缺陷改规范。工具/Skill可用性验证必须真调用一次确认，文件层+注册层双重校验，仅复制文件不算完成
18. **跨平台差异**：Windows 配置在 `%APPDATA%` 而非 `~/.config/`；敏感输出用 `sys.stdout.write()` 避免 BOM/CRLF 污染
19. **MCP自动识别**：图片/docx/pdf 直接调 convert_file，不绕路不废话
20. **edit_file 全量替换陷阱**：`edit_file` 的 `old_text` → `new_text` 是**全文所有匹配项替换**，非仅首项。模板化文件中不同分区的占位符可能相同，直接替换会导致跨分区污染。改进策略：① **优先追加**：定位到目标标题后追加新行，而非替换通用占位符；② **必须替换则用精准上下文**：将分区标题等唯一标识纳入 `old_text`；③ **写后验证**：写入后立即读取确认仅目标区被修改。
21. **原文润色原则**：用户说"帮我优化但保持原样"时——保留原始句式/语气（自嘲/苦涩/尖锐），保留叙事的松散结构，口语表达的"真"胜于书面表达的"美"，结尾情绪留白不强行下结论。判断标准：用户读着觉得"对，这就是我想的"。
22. **文档处理流水线范式**：产品/需求类文档的标准化处理流程 → 「原始格式(PDF/Docx/MD)→MD转换→AI润色→人工核实→`/plan` 结构化产出」。该范式适用于接口设计、产品规格、信息结构、PRD等需多重转换的场景。关键约束：AI润色后必须人工核实格式和语义保真度，`/plan` 前确保输入已冻结。
