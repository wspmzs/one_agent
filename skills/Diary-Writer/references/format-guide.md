# 格式规范

> 智能体执行写入时，用 `grep_search` 按类型名（如「工作待办」「知识链接」）定位对应段落，
> 只加载该类型的规则，不全量读取本文件。

---

# 工作待办

**写入分区**：`## ⛳工作待办`

**格式**：

```
- [ ] {标题 ≤30字}
>**截至日期**：{YYYY-MM-DD 或用户原始表述}
>**补充描述**：
>- {分条列点}
>- {目标清晰，步骤可循}
```

**规则**：

- 标题：必填，≤30字。从用户原话中提取或概括。
- 截至日期：用户未提供 → **必须追问**「这项任务的截至日期是什么时候？」
- 已完成识别：用户主动表示任务已完成（如「今天做了XX」「已经做完了」「完成了XX」「今天把XX弄好了」），或追问截至日期时用户回答已完成 → 截至日期写当天日期，勾选改为 `- [x]`，不再追问补充描述。
- 补充描述：仅当用户提供了标题以外的详细内容时才添加。分条列点——目标 → 步骤 → 备注。
- 严禁瞎编：不替用户做决定、不编造步骤、不添加用户没说的需求。
- 用户说得简单 → 写得简单；用户说得详细 → 写得详细。

**示例**：

```
- [ ] 输出Q3产品路线图初稿
>**截至日期**：2026-06-15
>**补充描述**：
>- 目标：输出Q3产品路线图初稿，对齐各业务线优先级
>- 步骤：收集各业务线需求 → 整理成优先级矩阵 → 评审修改
>- 备注：需要在6月中旬前完成初版供领导评审
```

```
- [x] 跟设计评审首页改版方案
>**截至日期**：2026-05-24
```

```
- [ ] {标题 ≤30字}

>**补充描述**：
>- {分条列点}

```

**规则**：

- 标题必填、截至日期缺失必问、补充描述按需。
- 已完成识别：用户主动表示已完成，或追问截至日期时回答已完成 → 截至日期写当天日期，勾选改为 `- [x]`，不再追问。

- 用户说得简单 → 写得简单；用户说得详细 → 写得详细。

```
- [ ] 预约牙科检查
>**截至日期**：下周
```

```
- [x] 咸鱼转卖消毒柜

# 知识链接

**写入分区**：`## 💩看了什么`

**有链接时**：

```
### [标题 ≤30字](URL)

#标签1 #标签2 #标签3

> ≤140字摘要

1. {知识要点}
2. {知识要点}
...
```

**无链接（长文本）时**：

```
### 标题 ≤30字

#标签1 #标签2 #标签3

> ≤140字摘要

1. {知识要点}
2. {知识要点}
...

```markdown
{原文内容}
```

**规则**：

- 标题：≤30字。有URL → 做超链接 `[标题](URL)`。无URL → 纯文本标题。
- 标签：最多3个，`#标签` 格式，选最能代表文章主旨的。不要强行凑出3个标签。
- 摘要：≤140字，独立段落，概括核心内容。
- 知识要点：编号列表。每条必须遵循 `references/hawkeye.md` 全部铁律。
- 原文：
  - **有URL → 不贴原文**（链接本身就是原文入口）
  - **无URL的长文本 → 用代码块贴原文**
- 有URL时：先用 `tavily_extract` 获取全文再提炼。
- 多条时：逐条处理，全部完成后再回复。

**示例（有链接）**：

```
### [RAG深度解析：分块、向量化、召回与重排](https://example.com)

#RAG #信息检索 #系统工程

> 从系统工程的视角拆解RAG全链路：文档解析→数据清洗→分块策略→向量化→召回→重排→TopK，用实际配置验证每个环节的工程权衡。

1. 当你想让一个系统记住海量信息并快速回答时，……。
2. 向量检索（RAG）结果不直接返回给用户，而是先……。
```

# 个人思考

**写入分区**：`## 🤫想了什么`

**格式**：

```
### {标题 ≤30字}

{优化后的正文}
```

**规则**：

- 标题：从用户思考中提取或概括，≤30字。
- 正文改写原则：
  - **100%保留原意不变**——绝不改变用户想表达的意思。
  - **改错别字和语病**。
  - **优化逻辑表达**——调整语序使行文流畅但不失原味。
  - **补充细节**——用户意思隐含但没说透的地方可适度补充，但严禁瞎编。
  - **专有名词**：首次出现时加英文注释：`专业名词（Technical terms）`。
  - **人称**：严格匹配用户原文——用户用第一人称你就用第一人称，用户用第三人称你就用第三人称。
  - **语气**：正式、精炼、专业的书面表达。

**示例**：

```
### 关于多智能体协作中上下文传递的思考

在多智能体协作体系（Multi-Agent Collaboration）中，上下文传递（Context Passing）是最容易被忽视的瓶颈。当一个智能体发起子任务时，如果不在消息中携带背景、目标和约束，接收方智能体就只能在信息不完整的情况下执行——这会导致返工、误解和效率下降。解决这一问题的关键在于建立结构化的通信协议：每条消息必须包含背景、目标和约束三个字段，且不允许省略其中任意一项。这不是技术问题，是纪律问题。
```
