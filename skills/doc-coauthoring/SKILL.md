# 文档与方案共创（Doc Co-Authoring）

本技能提供一套结构化的工作流，使你化身为引导者，与用户协作，将模糊、不完整甚至存在悖论的想法，逐步转化为清晰、可实施、无内在矛盾的结构化文档或方案。

你要主动扮演引导者，按需带着用户走完三个阶段：**需求理解与澄清 → 方案精炼与结构化 → 可实施性验证与盲点检测**。整个过程你需要始终保持对需求中缺失信息、逻辑矛盾和潜在错误的敏感，并善于通过提问帮助用户修正、补全自己的想法。

## 何时应主动提供此工作流

**触发条件：**

- 用户提到想要“写设计方案”“写落地方案”“写执行计划”“梳理需求”“头脑风暴”“制定旅行计划”“制定学习计划”等。
- 用户表现出思路尚不清晰，或需求可能存在冲突。
- 用户打算开始一项较复杂的、需要结构化规划的创作任务。

**初次提供：**
向用户提供一个三阶段的结构化共创流程，并简要说明每个阶段的目标：

1. **需求理解与澄清**：用户提供所有相关上下文，你通过提问挖掘缺失信息、发现逻辑矛盾，直到你对需求的理解足够深刻。
2. **方案精炼与结构化**：通过“提问→头脑风暴→筛选→起草→迭代修改”，逐节构建文档或方案。
3. **可实施性验证与盲点检测**：用一套独立视角（例如新的LLM会话）检验文档是否真的能让读者（包括执行者、审批者、另一个AI）准确理解，没有隐藏假设或矛盾。

向用户说明：这套流程能确保最终产出的文档在别人（或另一个AI）看来也是清晰、可用、无歧义的。询问用户是否愿意按此流程推进，还是更习惯自由对话。

如果用户拒绝，则回到自由对话模式。如果用户接受，自动进行“任务复杂度预评估”。

**任务复杂度预评估**
根据用户提供的信息，快速评估任务复杂度。满足以下 3 个条件中的任意 2 条，建议用户跳过完整流程，直接与LLM进行快速对话：

1. 用户已提供文档固定模板，且内容主要来自用户已知事实
2. 篇幅短小（预估 < 500 字）
3. 无逻辑矛盾或模糊决策点

如果用户仍希望使用完整流程，进入“阶段一：需求理解与澄清”。

## 阶段一：需求理解与澄清

**目标：** 弥合用户已知信息与你所知之间的鸿沟，让你能够像用户自己一样理解他的目标和约束，为后续的引导打下坚实基础。尤其要找出用户表述中缺失的背景信息、隐含的假设、可能存在的悖论，并把它们摆到桌面上。

### 第一步：收集元信息

首先，询问用户关于这份文档或方案的宏观背景：

1. 这是一份什么类型的文档？（例如：产品需求文档、技术方案、旅行计划、装修方案、学习计划）
2. 主要读者或受众是谁？（例如：开发团队、家人、客户、自己）
3. 这份文档需要带来的影响或决策是什么？
4. 是否有必须遵循的模板或格式？（鼓励用户提供模板、范例或链接）
5. 还有哪些约束需要提前知道？（时间、预算、技术栈、个人偏好等）

告诉用户，他们可以用任意方式回答，用短句、列表、甚至信息洪流的方式都可以，你会帮忙理清。

**如果用户提到了“模板”“文档类型”“写作风格”：**

- 检查 `references/templates/` 下是否有匹配的文档类型模板文件。如果有，向用户展示模板名称和结构，询问是否以此为蓝本开始共创。
  - **产品基准文档（PABD）**：位于 `references/templates/产品基准文档-PABD.md`。当用户说"产品基准文档""PABD文档"时，以此为模板。该模板面向零上下文启动的领域Agent编写，目标是尽可能全面地收集关于产品规划或设计的上下文信息。
- 检查 `references/style-guides/` 下是否有匹配的写作风格模板文件。如果有，用一小段话介绍写作风格。
- 如果模板不存在，询问用户能否提供。如果用户明确回答不能提供，按标准共创流程从零构建

**如果用户想基于已有的文档进行修改：**

- 首先读取文档当前内容。
- 留意文档中是否存在图像缺失替代文本，若有，说明当其他人使用AI理解此文档时可能无法看到图片，询问用户能否提供关于图片描述的文本。

### 第二步：鼓励信息倾泻

在获得基本元信息后，鼓励用户把所有能想到的上下文都倾倒出来。你可以这样说：

“现在你可以把关于这件事的所有想法、背景、担忧、参考、限制……一股脑儿全部倒给我，不必考虑组织。包括：为什么想做这件事、以前踩过哪些坑、为什么没选其他方案、相关方的态度、时间压力、技术限制等等。你也可以直接指向某个团队频道、文档链接或者文件，我帮你读取。”

**如果存在可用集成工具（如MCP服务器）：** 主动告知用户，你可以直接从Obsidian、飞书等处拉取上下文。如果当前环境未检测到集成，可建议用户启用相应的MCP服务器，或直接在对话中粘贴相关内容。

**如果用户提到未知实体或项目：** 先询问用户是否允许调用工具进行搜索来获得更多信息，不要擅自搜索。

在整个信息倾泻过程中，你要持续追踪已知与未知的差距，并为后面的澄清提问积攒素材。

### 第三步：澄清提问——找出漏洞、悖论与缺失信息

当用户发出“暂告一段落”的信号，或已经提供了大量内容后，基于你已掌握的信息，提出5-10个澄清问题。这些问题应当聚焦于：

- 逻辑上的不一致（例如：既要极简设计又要求大量数据展示，两者如何平衡？）
- 关键信息缺失（例如：预算范围、时间节点、目标用户画像）
- 隐含假设（例如：“前端自然会支持这个动效”——是否真的支持？）
- 潜在的矛盾需求（例如：要求离线可用却又依赖实时数据）
- 未言明的约束（例如：没有说但可能存在的技术栈限制、团队习惯）

将问题以编号列表呈现。告诉用户，他们可以简答（如“1:是，3:否，5:因为历史原因”），也可以继续倾倒信息，或者给你指出更多的资料链接。

**退出条件：** 当你发现你能够就边缘情况、取舍关系提出更深入的问题，而不再需要解释基础概念时，说明上下文已足够丰富。此时询问用户是否还有需要补充的内容，或是否进入下一阶段。

如果用户希望继续补充，则留在阶段一；否则，推进至阶段二。

## 阶段二：方案精炼与结构化

**目标：** 通过逐节构建的方式，将澄清后的需求转化为结构清晰、内容精要的文档或方案。每个小节都将经过“澄清焦点→头脑风暴→筛选→起草→迭代修改”的循环。

向用户说明接下来的做法：我们将一小节一小节地构建文档。每一节都会先询问他们想包含什么内容，然后你提供一系列可选方向，他们做筛选，最后你起草并不断精炼，直到满意。

### 确定文档结构

如果用户已经有了清晰的章节设想：

- 询问他们希望从哪一章节开始。建议从最核心、不确定性最高的章节开始（例如决策建议、技术方案核心、行程总览），而摘要/背景介绍等可以最后再写。

如果用户完全不知道应包含哪些章节：

- 基于文档类型和模板，提出3-5个合适的小节标题，并请用户确认或调整。

**结构确认后，立即创建带有占位符的文档骨架：**

- 使用 `write_file` 工具在 `D:\OneDrive\Obsidian\Y-头脑风暴`创建一个 Markdown 文件，文件名根据日期、文档主题和类型命名（如 `20260528-AI算法门户产品需求文档.md`、`20260601-青海湖大环线旅游攻略.md`）。
- 将所有章节标题写入文件，每个标题下暂时填入 `[待撰写]` 占位符。
- 创建完成后，告知用户文件的完整路径。

### 逐节推进——对每个小节执行以下6步

#### 1. 澄清焦点

宣布将开始撰写【章节名】。针对该章节，提出3-10个澄清性问题，挖掘用户对此部分的具体期望、必要的边界条件和潜在冲突。再次强调：发现任何前后不一致或逻辑漏洞时，要及时指出。

#### 2. 头脑风暴

根据该章节的复杂度，列出3-10个可能包含的要点。要点应涵盖：

- 用户已经提到的内容
- 基于第一阶段信息推断出但用户可能遗忘的角度
- 你可能察觉到的风险、权衡或替代方案

在列表末尾，可以询问是否需要更多选项。

#### 3. 筛选与组合

请用户指出哪些点保留、哪些删除、哪些合并，并可要求简要说明理由。举例：

- “保留 1,4,7,9”
- “删除 3（与1重复）”
- “删除 6（读者已经知道）”
- “合并 11 和 12”

如果用户给出的反馈是自由形式的（如“整体不错，但那个…”），而不是逐项编号，则根据上下文提取他的偏好并继续。

#### 4. 缺口检查

基于用户的选择，询问在此章节中是否还有什么重要的遗漏。

#### 5. 起草

用`edit_file`替换该章节的占位符为完整内容。告知用户章节已更新，并请他们通读后提出具体修改意见。强调他们无需直接编辑文档，只需告诉你哪里需要怎么改，例如：“删除X这一条，它已被Y覆盖”或“让第三段更简短”，这样你能更好地学习他们的风格偏好。

外部skill调用：

- 如果需要输出mermaid图表，先判断需要的图表类型，然后调用`mermaid-writing` SKILL

#### 6. 迭代打磨

根据用户的反馈，不断使用`grep_search`→`edit_file`进行修改。每次修改后都提示查看最新版本。如果用户自行修改了文档并让你阅读，你要默默记下这些改动反映的偏好，并在后续章节中应用。

当一个章节连续三轮没有实质性改动时，可提醒用户考虑是否还有可精简的内容。

**重复以上6步，直至所有章节均完成。**

### 收尾前的整体检查

当完成80%以上的章节后，主动通读整份文档，检查：

- 各部分之间逻辑流是否顺畅
- 是否存在前后矛盾、冗余表述
- 是否有“AI味儿”的抽象填充——即那些看似通顺但没有信息量的句子
- 每个句子是否都带有必要的重量

向用户提供你的观察，并根据反馈进行调整。

**所有章节完成后：** 再做最后一次整体审阅，确认一切妥当，然后询问是否进入第三阶段。

## 阶段三：可实施性验证与盲点检测

### 启动下游智能体审查

**目标：** 通过 `chat_with_agent` 将文档交给一个“零上下文”的独立视角——小美(id="designer"）——进行评审，以发现只有新鲜眼睛才能看到的模糊之处、错误假设和潜在实施问题——这用于模拟真实执行者或审核者第一次阅读文档的体验。

**要求**：

1. **预测读者问题** ：生成5-8个读者在试图理解或执行此文档时很可能会提出的真实问题。
2. **模拟答辩**： 提供`文件路径`及问题，看下游智能体回答得如何，汇总其理解正确与错误之处；
3. **同行评审**：要求下游智能体扫描文档中的模糊点、隐含假设、矛盾、缺失的前提知识。
4. **报告与修复：** 将发现的问题列表发给用户，说明“一个全新的视角指出了这些点可能存在误导或理解偏差”，然后回到阶段二的相关章节进行修复。

### 若无法调用其他智能体

将测试交给用户来完成。

1. **预测读者问题：** 生成5-8个潜在读者可能会提出的问题。
2. **提供测试方法：** 为用户生成简单的提示词（包含潜在用户的提问，扫描文档中的模糊点、隐含假设、矛盾、缺失的前提知识等要求），让用户自己开启全新对话，将提示词和文档内容粘贴进去，用于收集文档中的潜在问题。
3. **根据结果迭代：** 询问用户测试结果，哪里出了问题，然后针对性地回到阶段二进行修改。

### 退出条件

当一份全新的AI可以持续正确地回答问题、且未再指出新的模糊点或矛盾时，文档即告完备。

## 最终验收与交付

当验证通过后，告诉用户：

1. 建议他们自己再通读一遍，因为他们是文档的最终负责人。
2. 确认所有事实、链接、技术细节无误。
3. 反思文档是否真正达到了最初想要的影响。

**完成时，给出最后几点建议：**

- 可考虑在附录中对本次对话进行回顾，让读者看到方案是如何演进的。
- 善用附录为正文减重，同时又保留深度信息。
- 随着真实读者反馈的进入，持续迭代这份文档。

## 引导技巧与注意事项

- **语气：** 保持直接、有条理，少用推销式语言。解释理由时精练，核心是推进流程。
- **接纳偏离：** 如果用户想跳过某阶段，先确认是否真的跳过，并说明可能的风险。
- **处理挫败感：** 如果用户觉得过程太慢，表示理解，同时提出可以加速或精简步骤。
- **上下文维护：** 随时对未知信息保持好奇并提问，不要让盲区堆积。
- **艺术化管理：** 起草用`write_file`，修改用`edit_file`，每次修改后立即给出最新版本引用。不要将头脑风暴列表写入Artifact，它们只应留在对话中。
- **质量优先：** 每个迭代都要有实质性提升，不要为了赶进度而放过模糊点。最终目标是产出一份在真实世界里行得通的方案。
