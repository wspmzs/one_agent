# AGENTS.md 写入规范

> **映射文件：** AGENTS.md（各Agent工作区根目录）
> **核心原则：** 启动文件必须自包含。规则写进文件，不依赖外部引用。

## 定位

工作流程与协作规范。不含：沟通风格（SOUL.md）、用户画像（PROFILE.md）、记忆内容（MEMORY.md）。

## 元数据

头部必须包含元数据：

```yaml
---
summary: "Agent工作流程、规则与指南，每次会话必读"
read_when:
  - 手动引导工作区
  - 每次会话启动时
---
```

## 章节结构

### 1. 多Agent协作规范

**【位置规则】**
- 如用户使用多Agent协作模式，此章节必须放在最前面
- 如用户未使用协作模式，可删除此章节

**【强约束】必须写（如使用协作）：**

- 调度拓扑：唯一调度节点、星型通信
- 派发规则：什么任务→哪个Agent
- 通信方式：实时模式/后台模式
- 异常处理：反问机制、退回规则

**【核心约束】规则内联，少引用：**

- 协作协议的核心规则直接写入，不"> 详细见xxx.md"
- 每个Agent只写自己需要知道的规则
- 如果规则太长不适合内联，只写结论+操作指令

**【示例】**

```
## 多智能体协作

调度节点：小帅(impmzs)。流程：接收→规划→拆解→派发→汇总→审核→验收→交付。
通信：实时模式用于简单任务，后台模式用于复杂任务。
派发规则：前端/视觉→切图仔 | 重要任务/评审→小美 | 复杂脚本→程序猿 | 头脑风暴→七仔
异常：收到反问立即补充，无法解决→汇总问用户。
```

**【反例】不该写：**

- Agent能力边界表（只在SOUL.md写本Agent的能力）
- 通信命令的具体参数（平台实现细节，归工具层）

---

### 2. 记忆管理策略

**【强约束】必须写：**

- 文件分工：记忆文件/画像文件/日志各存什么
- 读取触发：什么情况读哪个文件
- 写入触发：什么事件触发写入
- 检索顺序：回答前搜什么、搜几次

**【示例】**

```
## 记忆管理
主动记，不等叫。偏好→画像文件，配置+决策→记忆文件，日志→日志目录。
回答历史问题前先搜记忆。
```

**【反例】不该写：**

- 具体的记忆条目格式（归MEMORY写入规范）
- 记忆内容本身（归记忆文件）

---

### 3. 常用 Skill、Tools、MCP 使用规范

**【强约束】必须写：**

- Skill清单：当前已安装列表，最多8个，按常用度排序
- Tools清单：常用内置工具，最多8个
- MCP服务：已配置的MCP工具（如有），最多8个
- 工具并行策略

**【格式约束】**

- 每类（Skill/Tools/MCP）最多8个
- 不标注匹配度（高/中/低等），无实际意义
- 不在文件中列出"建议禁用"项（禁用建议通过控制台会话告知用户）

**【自动更新】** keep-evolution优化时自动扫描：
- `<workspace>/skills/` 目录获取已安装技能
- `<workspace>/agent.json` 获取agent配置
- `<WORKING_DIR>/config.json` 获取MCP配置
- 从各SKILL.md前10行提取触发词/用途摘要
- 按Agent定位筛选常用项（最多8个），其余不写入

**【示例】**

```
## 常用 Skill、Tools、MCP 使用规范

**Skill：**
- self-review（输出检查，辅助）
- keep-evolution（记忆优化，核心）

**Tools：**
- read_file（文档检索）
- execute_shell_command（环境发现）
- memory_search（记忆检索）

工具并行：独立任务并行，依赖链串行。
```

**【反例】不该写：**

- Skill的具体使用方法（归各SKILL.md）
- MCP的详细配置格式（归记忆文件配置备忘）
- 不相关/禁用的工具（控制台会话告知用户）

---

### 4. 绝对禁忌

**【强约束】必须写：**

- 安全红线：禁止泄露的数据类型
- 操作边界：需确认的操作、完全禁止的操作
- 兜底规则：不确定时怎么办

**【格式】** `-` 短句，不加解释。越短越好。

**【示例】**

```
## 安全
- 绝不泄露私密数据。
- 系统级操作先问再执行。
- 拿不准先确认。
```

---

### 5. 项目路径规范

**【强约束】必须写：**

- 统一项目根目录路径
- 标准目录结构：README.md / 01-requirements/ / 02-design/ / 03-code/ / 04-tests/ / 05-others/
- README.md必建规则

**【示例】**

```
## 项目路径
所有协作任务统一存放在项目根目录下。
标准结构：README.md / 01-requirements/ / 02-design/ / 03-code/ / 04-tests/ / 05-others/
任务发起者第一时间创建项目文件夹和README。
```

---

## 强约束（不写什么）

**绝对不写：**

- 沟通风格、语气要求（归SOUL.md）
- 用户画像、技术能力边界（归PROFILE.md）
- 具体执行手法、工作方式（归SOUL.md）
- 记忆内容本身（归MEMORY.md）
- 外部文件引用链（"> 详细见xxx.md"、"见xxx第几章"）
- 协议详细字段的冗长定义（写结论+操作指令即可）

**格式限制：**

- 纯Markdown文本
- 不用流程图（prompt渲染路径不可控）
- 不用mermaid图、ASCII码
- 代码块不超过5行
- 列表优于表格（内容量小，列表更紧凑）
- 不使用表情
