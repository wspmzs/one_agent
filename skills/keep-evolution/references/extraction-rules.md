# 信息提取与写入位置判断规则

> **用途：** keep-evolution技能从用户指定文件提取信息时，判断内容类型和写入位置。

## 信息分类流程

```
用户输入内容/文件路径
        ↓
这段内容是什么类型？
        ↓
┌─────────────────────────────────────────────────────────┐
│ 内容类型                    → 写入位置                    │
├─────────────────────────────────────────────────────────┤
│ Agent行为规范/执行流程       → 该Agent的SOUL.md/AGENTS.md  │
│ 实践经验/踩坑记录            → MEMORY.md                 │
│ 用户偏好/习惯               → PROFILE.md                 │
│ 专有名词/术语定义            → MEMORY.md专有名词章节      │
│ 配置信息/工具设置            → MEMORY.md领域知识章节      │
│ 工作流程/SOP步骤             → MEMORY.md工作流程章节      │
└─────────────────────────────────────────────────────────┘
```

## 判断标准

| 问题 | 答案 | 写入位置 |
|------|------|----------|
| 这是某个Agent特有的行为准则吗？ | 是 → 该Agent的SOUL.md | 否 → 继续判断 |
| 这是某个Agent的执行流程吗？ | 是 → 该Agent的AGENTS.md | 否 → 继续判断 |
| 这是用户偏好/习惯吗？ | 是 → PROFILE.md | 否 → 继续判断 |
| 这是实践经验/踩坑记录吗？ | 是 → MEMORY.md | 否 → 继续判断 |
| 这是配置信息/工具设置吗？ | 是 → MEMORY.md领域知识章节 | 否 → 继续判断 |
| 这是专有名词/术语定义吗？ | 是 → MEMORY.md专有名词章节 | 否 → 继续判断 |
| 这是工作流程/SOP步骤吗？ | 是 → MEMORY.md工作流程章节 | 否 → MEMORY.md（按内容归类） |

## 提取来源与处理方式

### 1. 文本内容

**来源：** 用户直接粘贴的文本

**处理：**
1. 识别内容类型（按上述判断标准）
2. 提取关键信息，去除冗余
3. 转换为规范格式（短句、列表）
4. 写入对应文件的对应章节

### 2. 文件路径

**来源：** 用户提供的文件路径（如 `C:\Users\xxx\project\README.md`）

**处理：**
1. 读取文件内容
2. 识别内容类型
3. 提取有价值信息
4. 写入对应文件

### 3. 每日记忆文件

**来源：** `memory/YYYY-MM-DD.md`

**处理：**
1. 读取指定日期的记忆文件
2. 筛选有价值的经验/教训/配置
3. 去重（与MEMORY.md现有内容对比）
4. 写入MEMORY.md对应章节

### 4. 对话记录

**来源：** 用户提供的对话记录文本或文件

**处理：**
1. 识别对话中的关键决策、经验、偏好
2. 按内容类型分类
3. 精简为短句/列表格式
4. 写入对应文件

## 写入格式要求

### 通用格式

- 短句，一.idea per line
- 不使用表情
- 不使用复杂长句
- 不过多空行

### MEMORY.md 格式

- 经验条目：`### 经验名称（日期）`，正文3-5行
- 配置条目：代码块（不超过5行）
- 专有名词：`- **术语：** 一句话定义`

### PROFILE.md 格式

- 用户信息：`- **关键词：** 内容`
- 技术边界：表格形式

### AGENTS.md / SOUL.md 格式

- 规则条目：`- 短句规则`
- 行为准则：粗体标题句 + `-` 子句展开

## 去重规则

写入前必须检查目标文件是否已存在相似内容：

1. **完全相同：** 跳过，不写入
2. **高度相似（>80%重叠）：** 合并为一条，保留更完整的版本
3. **部分重叠：** 补充新信息，不重复已有内容
4. **完全不同：** 新增条目

## 规则附原因（Reason-Anchored Rule）

写入规则时，不能只写"能做什么/禁止做什么"，必须用简单语言附上原因。

**格式：**
- 禁令格式：「禁止xxx，因为xxxx。」
- 许可格式：「你可以xxx，因为xxxx。」

**原因：** 没有原因支撑的规则，Agent在遇到规则未覆盖的情况时会自行推理出偏离预期的结论。附上原因能让Agent理解规则背后的价值观。
